# DBtest1

1. 엑셀 vs 데이터베이스 (테이블 분할의 이유)
•	엑셀과 다른 점: 엑셀은 하나의 시트에 모든 데이터를 때려 박아 넣기 때문에 데이터가 중복되고, 오타가 나기 쉽고, 수십만 건이 넘어가면 느려집니다. 데이터베이스(RDB)는 엄격한 규칙(제약조건)과 관계를 통해 데이터의 정확성(무결성)을 보장합니다.
•	테이블을 나누는 이유: 예를 들어 회원 한 명이 책을 5번 빌렸다고 해서, 회원 테이블에 그 사람의 이름과 전화번호를 5번 반복해서 적으면 저장 공간이 낭비되고 전화번호가 바뀔 때 5곳을 다 수정해야 합니다. 이를 방지하기 위해 [회원 테이블]과 [대여 테이블]을 따로 나누어 저장합니다.


3. 관계와 키 (PK, FK, 1:N)
•	PK (Primary Key, 기본키): 테이블에서 각 행(데이터)을 유일하게 식별할 수 있는 고유 번호입니다. (예: 학번, 주민번호, 회원번호). 중복될 수 없고, 비어있을 수(NULL) 없습니다.
•	FK (Foreign Key, 외래키): 다른 테이블의 PK를 참조하는 연결 고리입니다. 예를 들어 대여 테이블에 회원번호를 적어두면, 이 회원번호가 바로 FK가 됩니다.
•	1:N (일대다) 관계: 한쪽 데이터 하나가 반대쪽 데이터 여러 개와 연결되는 구조입니다.
o	예시: 국가(1) : 시민(N) (한 국가는 여러 시민을 가지지만, 시민은 하나의 국가에 속함)
o	과제 팁: 1:N 관계에서는 항상 'N' 쪽에 FK가 들어갑니다. (시민 테이블에 국가ID를 적음)


5. 제약조건 (Constraints)
데이터베이스에 쓰레기 데이터가 들어오지 못하게 막는 방어벽입니다.
•	NOT NULL: "이 칸은 절대 비워두면 안 돼!" (예: 회원 가입 시 비밀번호, 이름 등)
•	UNIQUE: "중복된 값은 절대 안 돼!" (예: 로그인 아이디, 이메일, 전화번호 등)
•	참조 무결성 (FK 제약): 회원 테이블에 없는 회원번호 999번이 대여 테이블에 저장되는 것을 데이터베이스가 알아서 에러(Error)를 내며 막아주는 규칙입니다.


6. 데이터 조작 명령어 (CRUD)
•	INSERT (생성): 테이블에 새로운 행(데이터)을 집어넣을 때 씁니다. (회원가입, 주문하기)
•	SELECT (조회): 저장된 데이터를 가져와서 볼 때 씁니다. (마이페이지 조회, 상품 검색)
•	UPDATE (수정): 기존에 있던 데이터를 바꿀 때 씁니다. (비밀번호 변경, 성별 수정)
•	DELETE (삭제): 데이터를 지울 때 씁니다. (회원 탈퇴, 글 삭제)


8. 정렬, 제한, 집계 (SQL의 기본 스킬)
•	WHERE: 데이터를 내 입맛대로 필터링(조건)할 때 씁니다. (WHERE 성별 = '남')
•	ORDER BY: 데이터를 정렬할 때 씁니다. 기본은 오름차순(ASC), 내림차순은 DESC를 붙입니다. (ORDER BY 가입일 DESC)
•	LIMIT: 데이터가 너무 많을 때 상위 몇 개만 끊어서 볼 때 씁니다. (LIMIT 10)
•	GROUP BY: 데이터를 특정 컬럼 기준으로 그룹 묶기를 할 때 씁니다. (예: '성별'로 묶어서 남자는 몇 명, 여자는 몇 명인지 계산할 때)
•	집계 함수: * COUNT: 데이터 개수 세기
o	SUM: 합계 구하기
o	AVG: 평균 구하기


10. 조인 (JOIN) - 쪼개진 테이블 합치기
과제 목표 중 "연결된 데이터를 한 번에 뽑는 방법"에 해당합니다.
•	INNER JOIN (내부 조인): 두 테이블 양쪽에 모두 존재하는 일치하는 데이터만 합쳐서 보여줍니다. (예: 구매 이력이 있는 회원만 보기)
•	LEFT JOIN (외부 조인): 왼쪽 테이블의 모든 데이터를 보여주고, 오른쪽 테이블에 매칭되는 데이터가 없으면 NULL로 채워서 보여줍니다. (예: 구매를 한 번도 안 한 사람을 포함해서 전체 회원 목록 보기)



11. 서브쿼리 (Subquery)
•	정의: 쿼리문 안에 또 들어있는 쿼리문입니다.
•	이해: 서브쿼리는 "택배 보낼 주소를 먼저 찾고(서브쿼리), 그 주소로 택배를 보낸다(메인쿼리)"처럼 단계별로 문제를 풀 때 사용합니다.
•	주의점: 바로 이전 질문에서 겪으셨던 것처럼, MySQL에서는 UPDATE나 DELETE 시 동일한 테이블을 서브쿼리로 직접 참조하면 1093 에러가 나므로 우회하거나 단순화해야 합니다.


13. 인덱스 (Index) - 책의 찾아보기(색인)
•	왜 필요한가: 책에서 특정 단어를 찾을 때 첫 페이지부터 끝까지 다 읽으면(Full Scan) 너무 오래 걸립니다. 뒤쪽에 있는 '찾아보기(인덱스)'를 보면 몇 페이지에 있는지 바로 알 수 있죠. DB도 똑같습니다.
•	어디에 적용하나: WHERE 절에 자주 쓰이는 검색 조건 컬럼, 정렬(ORDER BY)에 자주 쓰이는 컬럼, 혹은 JOIN할 때 연결고리가 되는 컬럼에 적용하면 조회 속도가 획기적으로 빨라집니다.
인덱스는 목차와 같이 특정 값을 빠르게 찾기 위한 자료 구조

자료형(Data Type, 데이터 타입)이란 데이터베이스나 프로그래밍에서 "이 칸(컬럼)에는 어떤 종류의 데이터만 집어넣을 수 있는지" 컴퓨터에게 미리 알려주는 규칙이자 데이터의 상자 크기


•  엑셀 vs DB 및 테이블 분할 이유
"엑셀은 데이터가 중복되어 꼬이기 쉬운 평면 구조인 반면, DB는 중복과 오타를 막기 위해 데이터를 주제별 테이블로 쪼개어 정교하게 관리합니다."
•  PK/FK 및 1:N 관계
"PK는 고유한 데이터 번호표고 FK는 다른 테이블의 PK를 가리키는 연결고리이며, 이 둘을 통해 부모(1)와 자식(N)처럼 데이터를 유기적으로 엮어줍니다."
•  SELECT / INSERT / UPDATE / DELETE (CRUD) 구분
"데이터를 조회할 때는 SELECT, 새로 추가할 때는 INSERT, 수정할 때는 UPDATE, 삭제할 때는 DELETE를 사용합니다."
•  JOIN과 GROUP BY의 연계
"JOIN으로 찢어져 있던 여러 테이블을 옆으로 이어 붙인 뒤, GROUP BY를 사용해 원하는 컬럼끼리 묶어서 한눈에 통계를 냅니다."
•  실무 요구사항 처리 (검색/정렬/집계/랭킹)
"검색은 WHERE, 정렬은 ORDER BY, 집계는 SUM/AVG 같은 함수를 쓰며, 순위(랭킹)는 특정 기준 정렬 후 LIMIT으로 끊거나 RANK() 함수로 해결합니다."
•  인덱스의 필요성과 적용 컬럼
"인덱스는 책 뒤쪽의 찾아보기(목차)와 같아서 전체를 다 뒤지는 디스크 접근을 줄여주며, 주로 WHERE나 JOIN 조건에 단골로 쓰이는 컬럼에 적용해야 효과적입니다."
